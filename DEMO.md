# MicroPKI Demo Walkthrough

Подробное описание автоматизированного демо-скрипта `demo/demo.py`.

## Как запустить

```bash
# Из директории micropki/
python demo/demo.py

# Или через CLI
python -m micropki demo run
```

Скрипт создаёт временную директорию через `tempfile.mkdtemp()` и разворачивает
там полную PKI-инфраструктуру. Каждый запуск — чистое окружение, никаких следов
от предыдущих запусков.

---

## Шаги и их смысл

### Шаг 1 — Инициализация Root CA

**Что делает:** создаёт самоподписанный Root CA с RSA-4096 ключом, сроком 10 лет,
записывает в SQLite БД.

**Почему важно:** Root CA — вершина доверия в PKI. Его приватный ключ должен быть
максимально защищён (в реальности хранится в HSM оффлайн). Алгоритм RSA-4096
выбран потому что корневые сертификаты живут долго и должны устоять перед будущими
угрозами.

**Выходные файлы:**
- `pki/certs/ca.cert.pem` — публичный сертификат Root CA
- `pki/private/ca.key.pem` — зашифрованный приватный ключ (AES-256-CBC + PBKDF2)

---

### Шаг 2 — Выпуск Intermediate CA

**Что делает:** Root CA подписывает сертификат Intermediate CA с `pathlen=0`
(нельзя создать CA ниже этого уровня).

**Почему важно:** двухуровневая иерархия — стандартная практика. Root CA остаётся
оффлайн, Intermediate CA используется для повседневного выпуска. Если Intermediate
скомпрометирован — отзываем только его, Root CA не трогаем. `pathlen=0` — техническое
ограничение: никто не сможет создать ещё один CA с этим сертификатом.

---

### Шаг 3 — Выпуск server-сертификата

**Что делает:** Intermediate CA выпускает server-сертификат с SAN
(`dns:demo.local`, `dns:localhost`, `ip:127.0.0.1`), шаблон `server`.

**Почему важно:** браузеры и TLS-клиенты проверяют SAN (Subject Alternative Names),
а не CN. Без корректного SAN сертификат не пройдёт верификацию хоста. Шаблон `server`
устанавливает нужные расширения: `serverAuth` в EKU, `digitalSignature` + `keyEncipherment`
в KeyUsage.

---

### Шаг 4 — Выпуск client-сертификата

**Что делает:** выпускает сертификат для клиентской аутентификации с email SAN.

**Почему важно:** mTLS (mutual TLS) требует клиентского сертификата. Шаблон `client`
устанавливает `clientAuth` в EKU. Email в SAN используется для привязки сертификата
к конкретному пользователю.

---

### Шаг 5 — Выпуск OCSP responder-сертификата

**Что делает:** выпускает специальный сертификат с расширением `OCSPSigning` в EKU,
который будет использоваться OCSP-респондером для подписи ответов.

**Почему важно:** OCSP-ответы должны быть подписаны. Использование отдельного
ключа для OCSP (а не ключа CA) — best practice: если OCSP-сервер скомпрометирован,
CA-ключ остаётся в безопасности. Сертификат выпускается с коротким сроком для
ограничения ущерба.

---

### Шаг 6 — Выпуск code-signing сертификата

**Что делает:** выпускает сертификат для подписи кода с `codeSigning` в EKU.

**Почему важно:** специализированные EKU позволяют ограничить область применения
сертификата. Code signing сертификат нельзя использовать для TLS-сервера и наоборот
(если клиент проверяет EKU). Используется в шаге 14 для демонстрации подписи файлов.

---

### Шаг 7 — Валидация server-сертификата

**Что делает:** запускает `client validate` с режимом `chain` — проверяет цепочку
подписей от leaf до Root CA.

**Почему важно:** демонстрирует, что PKI работает корректно end-to-end. Проверяются:
подписи на каждом уровне, сроки действия, BasicConstraints (CA=True у промежуточных),
pathLenConstraint.

---

### Шаг 8 — Генерация CRL

**Что делает:** генерирует Certificate Revocation List (CRL) от Intermediate CA.

**Почему важно:** CRL — один из двух стандартных механизмов проверки отзыва (второй —
OCSP). Содержит список серийных номеров отозванных сертификатов. Клиенты периодически
скачивают CRL и проверяют сертификаты по нему.

---

### Шаг 9 — Отзыв server-сертификата

**Что делает:** отзывает server-сертификат с причиной `superseded` (заменён новым).

**Почему важно:** отзыв критически важен при компрометации ключа, смене владельца
или истечении срока до официальной даты. Причина отзыва (Reason Code) помогает
клиентам понять что произошло и как реагировать.

---

### Шаг 10 — Проверка отозванного сертификата

**Что делает:** проверяет что статус сертификата в БД изменился на `revoked`.

**Почему важно:** демонстрирует что отзыв действительно фиксируется в системе.
В реальном PKI клиенты получают эту информацию через CRL или OCSP перед каждым
TLS-соединением.

---

### Шаг 11 — Проверка целостности аудит-лога

**Что делает:** запускает `audit verify` — проходит по всей цепочке SHA-256 хешей
в NDJSON-логе и проверяет что ни одна запись не была изменена или удалена.

**Почему важно:** аудит-лог должен быть tamper-evident. Хеш-цепочка (каждая запись
содержит хеш предыдущей) позволяет обнаружить любую модификацию задним числом —
аналог blockchain для журнала событий.

---

### Шаг 12 — Проверка политики

**Что делает:** пытается создать Root CA с RSA-2048 ключом — этот шаг должен
завершиться с ошибкой (политика требует минимум RSA-4096 для CA).

**Почему важно:** политики безопасности — первая линия обороны. Автоматизированная
проверка параметров при выпуске предотвращает случайное создание слабых сертификатов.

---

### Шаг 13 — TLS демо

**Что делает:** запускает минимальный Python HTTPS-сервер с нашим server-сертификатом,
затем подключается к нему как TLS-клиент, используя Root CA как доверенный якорь.

**Почему важно:** демонстрирует что сертификаты действительно работают для TLS.
Клиент проверяет: подпись цепочки, срок действия, CN/SAN соответствие хосту.
Всё это происходит автоматически через стандартный Python `ssl` модуль.

---

### Шаг 14 — Code signing демо

**Что делает:** подписывает Python-скрипт приватным ключом code-signing сертификата
через `client sign`, затем верифицирует подпись через `client verify`.

**Почему важно:** демонстрирует сценарий подписи кода — распространённый в CI/CD.
Detached-подпись (отдельный `.sig` файл) позволяет дистрибутировать подпись отдельно
от файла. Верификация проверяет и криптографическую корректность подписи, и валидность
сертификата.

---

### Шаг 15 — Итоговая статистика

Выводит сводку: сколько шагов прошло, сколько упало, список провалившихся шагов.
Скрипт возвращает код выхода `0` если все шаги прошли, `1` если есть провалы.

## Структура временной директории после демо

```
/tmp/micropki_demo_XXXXXXXX/
├── secrets/
│   ├── root.pass          # случайная парольная фраза Root CA
│   └── intermediate.pass  # случайная парольная фраза Intermediate CA
├── pki/
│   ├── certs/
│   │   ├── ca.cert.pem              # Root CA сертификат
│   │   ├── intermediate.cert.pem   # Intermediate CA сертификат
│   │   ├── demo.local.cert.pem     # Server сертификат
│   │   ├── Demo_Client.cert.pem    # Client сертификат
│   │   └── Demo_Code_Signer.cert.pem  # Code signing сертификат
│   ├── private/
│   │   ├── ca.key.pem               # Зашифрованный Root CA ключ
│   │   └── intermediate.key.pem    # Зашифрованный Intermediate CA ключ
│   ├── audit/
│   │   └── audit.log               # NDJSON аудит-лог
│   ├── micropki.db                 # SQLite база сертификатов
│   └── intermediate.crl.pem       # Certificate Revocation List
├── script.py                       # Тестовый файл для code signing
├── script.py.sig                   # Detached подпись
└── server_chain.pem               # Цепочка для TLS (leaf + intermediate)
```

## Интеграция в CI/CD

Демо-скрипт можно включить в pipeline для smoke-тестирования PKI:

```yaml
# .github/workflows/demo.yml
- name: Запустить PKI демо
  run: python demo/demo.py
  working-directory: micropki/
```

Скрипт возвращает ненулевой код выхода при любом провале — CI поймает это как failure.
